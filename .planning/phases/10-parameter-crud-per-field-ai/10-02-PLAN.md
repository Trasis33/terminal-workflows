  <action>
Extend `ParamEditorModel` with full field editing capabilities.

**Expanded param fields — when a param row is expanded, show editable fields:**

1. **Name field** (textinput) — already exists from Plan 01. Enhance:
   - On every keystroke in the name field, emit a `paramRenamedMsg{oldName, newName}` tea.Msg.
   - Validate: if the new name matches any other param's name, show inline error "Parameter name already exists" in red below the input. Do NOT block navigation — only block Save per CONTEXT.md decision.
   - Empty name shows inline error "Parameter name is required".

2. **Type selector** — new inline selector widget:
   - Display as: `Type: [text] enum  dynamic  list` where current type is highlighted/bracketed.
   - Left/Right arrows or h/l change type when this field is focused.
   - **Soft staging per CONTEXT.md:** When changing type (e.g., enum→text), do NOT clear the enum options from `paramEntry`. Keep all metadata in memory during the editing session. Only on Save, if type is incompatible with metadata, show a warning message: "Saving will clear enum options for '{name}' (now type text). Continue?" with options to Save, Discard, Keep Editing.
   - The type field is always visible in the expanded view.

3. **Default value field** (textinput) — PCRD-05:
   - Always visible when expanded for text, enum, and dynamic types.
   - For enum type, the default should be one of the options (show warning if not, but don't block).

4. **Enum options field** (textinput) — PCRD-06:
   - Only visible when type is "enum".
   - Comma-separated input: "opt1, opt2, opt3" stored as `[]string`.
   - Prefix an option with `*` to mark it as default: "*opt1, opt2, opt3".
   - Show current options rendered as pills/tags below the input for visual feedback.

5. **Dynamic command field** (textinput) — PCRD-07:
   - Only visible when type is "dynamic".
   - Full-width input for the shell command.
   - Placeholder: "e.g., git branch --list"

**Field navigation within expanded param:**
- Tab moves: name → type → default → (enum options OR dynamic command, depending on type) → next param/add button.
- Shift+Tab reverses.
- Add `focusedSubField int` to `paramEntry` or to `ParamEditorModel` to track which sub-field within the expanded param has focus.

**Live command template preview (PCRD-03):**
- When a param is renamed, emit `paramRenamedMsg` from `ParamEditorModel.Update()`.
- In `FormModel.Update()` (form.go), handle `paramRenamedMsg`:
  - Replace all `{{oldName...}}` patterns with `{{newName...}}` in the command textarea value.
  - Use `strings.ReplaceAll` with regex to handle `{{old_name}}`, `{{old_name:default}}`, `{{old_name|opt1|opt2}}`, `{{old_name!cmd}}` patterns.
  - The command textarea should update visually in real-time as the user types the new name.

**Type change message:**
- Define `paramTypeChangedMsg{index int, oldType, newType string}` — emitted when type selector changes.
- FormModel can use this to update command template syntax if needed (e.g., if changing from enum to text, the `{{name|opt1|opt2}}` pattern in the command could be simplified to `{{name}}` — but per soft staging, this only happens on save).

**Save-time validation (form.go):**
- Before saving, iterate params. For each param where type is incompatible with stored metadata:
  - text type with non-empty `options` or `dynamicCmd` → warn.
  - enum type with empty `options` → warn.
  - dynamic type with empty `dynamicCmd` → warn (but allow, user might fill it later).
- Show a confirmation dialog if warnings exist. If user confirms, strip incompatible metadata before persisting.
- If no warnings, save directly.

**Important anti-patterns to avoid:**
- Do NOT auto-modify the command template on type change (only on rename). Type change soft staging means the command template retains its original syntax until explicit save.
- Do NOT block field navigation on validation errors. Inline errors are informational; only Save is gated on critical errors (empty name).
